架构设计背景与方法论

1.1 TOGAF ADM 方法论说明

本报告采用 TOGAF 9.2 架构开发方法（Architecture Development Method，ADM）作为企业架构设计的核心方法论。TOGAF ADM 是一套经过全球数千个企业架构实践验证的迭代式架构开发方法，由8个相位（Phase）和一个持续迭代的需求管理中心构成，确保企业架构从战略愿景到技术落地的全程可控。

本项目在 TOGAF ADM 框架下，重点关注以下四个核心相位，形成四层架构体系：

TOGAF ADM 相位与本项目映射关系

ADM 相位中文名称本项目输出完成时间
Phase A架构愿景（Architecture Vision）转型愿景、架构驱动因素、架构原则、利益相关方分析2026年6月（本报告第二章）
Phase B业务架构（Business Architecture）业务能力地图（L1-L3）、端到端流程设计（OTC/PTP/MTC）2026年7月（本报告第三章）

TOGAF ADM 中有两个关键概念在本报告中频繁使用：架构构建块（Architecture Building Blocks，ABBs）指按照架构标准定义的能力或功能单元，例如"主数据管理能力"；解决方案构建块（Solution Building Blocks，SBBs）指满足 ABB 的具体实现方案，例如"SAP MDG 模块"。本报告在 L2-L3 层级完成从 ABB 到 SBB 的映射，为 SAP S/4HANA 实施提供明确的架构指引。

1.2 架构设计原则（12条）

架构设计原则是指导所有架构决策的核心准则，优先级高于具体的技术选型和实施方案。以下12条原则经过佛照项目团队与艾维登架构团队联合确认，适用于本项目所有架构设计工作。

通用原则（4条）
#原则名称描述对S/4HANA的影响
GP-1业务主导架构决策以业务价值为出发点，技术服从业务，不为技术而技术Greenfield方法论优先采用SAP标准流程，减少不必要定制化开发
GP-2数据驱动数据是集团最重要的战略资产，所有系统以数据质量和数据可见性为核心设计约束MDG作为主数据治理中枢，全业务域数据流向S/4HANA统一汇聚
GP-3最大化标准化优先采用SAP S/4HANA标准功能，自定义开发（Z程序）须经架构评审委员会批准，最小化定制范围减少升级负担，ECC现有Z程序须经评估后按"小改/中改/重写/废弃"分类处理
GP-4云原生优先优先采用SAP Business Technology Platform（BTP）上的云原生扩展，而非传统ABAP核心定制扩展开发部署在BTP侧扩展（Side-by-Side Extension），保持S/4HANA核心整洁
业务架构原则（2条）
#原则名称描述
BA-1端到端流程完整性业务流程设计必须覆盖完整的端到端价值链（从客户需求到客户收款、从供应商询价到货款支付、从生产计划到成本归集），不允许存在流程断点或数据孤岛，尤其是跨系统的流程衔接
BA-2集团管控统一性19家法人必须使用统一的流程模板和数据标准，允许在全局模板基础上做有限的法人级配置差异，但不允许创建差异性子流程导致合并报表困难。泰国（FZ40）、德国（FZ33）法人须支持多语言、多币种、本地合规要求
数据架构原则（2条）
#原则名称描述
DA-1单一可信数据源（Single Source of Truth）每类主数据只有一个权威系统（System of Record）维护，其他系统只读使用，不允许在多个系统中各自独立维护导致数据不一致。物料主数据的 SOR 为 S/4HANA MDG，通过 CPI 向 WMS/MES/PLM 等同步
DA-2数据生命周期管理所有主数据必须有明确的生命周期状态（创建/审核/启用/停用/归档），由 MDG 工作流控制，禁止绕开审批流程直接修改生产系统主数据
应用架构原则（2条）
#原则名称描述
AA-1松耦合集成所有系统间集成必须通过 SAP CPI（Integration Suite）进行，禁止系统间点对点直连（Point-to-Point）。这确保了集成逻辑的集中管控、监控和变更控制，降低系统替换时的影响范围
AA-2API 优先新开发的集成接口优先采用 RESTful API 或 OData 协议，对于历史系统采用 IDoc 或 BAPI 的接口可保留但需在 CPI 中封装，对外统一暴露 API 接口
技术架构原则（2条）
#原则名称描述
TA-1安全设计内置安全性不是事后补充，而是从架构设计阶段内置。包括：最小权限原则（用户只拥有工作所需的最低权限）、职责分离（Segregation of Duties，SoD）、关键操作审计日志、数据传输加密（TLS 1.3）
TA-2高可用与容灾S/4HANA PCE 生产系统 RTO（恢复时间目标）≤4小时，RPO（恢复点目标）≤1小时。SAP负责PCE平台层的高可用，佛照IT负责网络接入和终端侧的可用性

架构愿景（Architecture Vision — TOGAF Phase A）

2.1 架构驱动因素（Business Drivers）

基于管理层访谈（董事长余中民、总经理张学权、财务负责人曾肖静等7位高管）、175条业务调研问题和佛照2025年年报数据分析，识别出以下六大架构驱动因素：

#驱动因素业务背景架构响应优先级
BD-1全球化合规需求泰国FZ40工厂启动运营，ECC不支持泰国本位币（泰铢）记账、不支持泰国增值税（7%）和预提税申报；德国FZ33需符合IFRS和EU税务合规；海外收入占比20.8%且仍在增长S/4HANA PCE原生支持多币种、多语言、多税务体制；DRC模块支持电子发票和泰国税务申报；Group Reporting实现多准则合并高
BD-2集团管控低效19家法人收购后系统各自独立，合并报表靠系统外手工抵消，月结需7-8个工作日；经营分析会数据来自各部门手工汇总，存在口径不一致、数据失真风险S/4HANA统一集团平台；Group Reporting实现自动合并抵消；MDG确保跨法人数据口径一致高
BD-3主数据混乱导致运营断点物料主数据多部门分散维护，无统一管理平台；存在一物多码、一客多码；MRP因BOM维护不及时导致跑不起来（生产计划完全依靠手工）；每年新增物料编码规避考核，截至2025年已积累6.2万个物料主数据SAP MDG作为主数据治理中枢，统一物料/BP/财务主数据的创建、变更、停用全生命周期；建立主数据评审委员会高
BD-4系统性能制约ECC系统上线已17年，每年产生仅2024年就达430万出入库凭证、110万销售订单；物料账跑批需7-8小时；关闭订单2-3小时；频繁出现程序异常导致财务数据错误S/4HANA基于HANA内存数据库，核心操作性能提升10-100倍；月结关账目标从7天→2天高
BD-5研产供销协同断裂设计BOM→制造BOM→成本BOM转换靠手工；MES/WMS与SAP无完整集成（仅部分覆盖）；销售预测准确率30-40%；定价审批OA流程7-8个环节、耗时过长；SAP与WMS/MES"两张皮"Greenfield全新架构打通PLM→SAP PP→MES→EWM完整数字链；CPI实现实时集成；SD启用标准定价条件方案替代OA审批中
BD-6盈利能力持续恶化2025年扣非净利率仅0.95%（行业底部）；经营现金流同比暴跌50%；成本核算粗放（标准成本系统外估算）；库存管理不透明（看不清事业部/渠道归属）；研发费用加计抵扣无法正确统计CO模块规范成本核算；启用利润中心和段实现多维盈利分析；IBP（长期）优化库存和计划协同中
2.2 架构愿景陈述与利益相关方

目标架构愿景（Target Architecture Vision Statement）：

以 SAP S/4HANA PCE 为数字化核心底座，以 SAP MDG 为主数据治理中枢，以 SAP CPI 为集成神经网络，构建覆盖佛照集团19家法人、贯通研产供销财全价值链的统一智造运营平台。通过数据标准化、流程标准化、管控一体化，实现从"分散管理的多元化集团"向"数据驱动的智造型集团"的跨越，支撑佛照在全球照明与汽车灯具市场的核心竞争力持续提升，2027年Q1目标上线。

量化目标：月结关账天数 7天→2天 | 存货周转天数 109天→75天 | 应收周转天数 90天→70天 | 主数据准确率 <70%→99% | 合并报表出具效率提升60% | MRP计划有效执行率 <30%→85%

数据架构（Data Architecture — TOGAF Phase C Part 1）

4.1 L1：数据治理战略与框架

数据架构是佛照本次数字化转型的核心基础设施。诊断阶段发现，佛照当前数字化水平低下的根本原因在于主数据混乱和数据标准缺失，这导致MRP无法运行、合并报表手工、成本核算不准确等一系列连锁问题。因此，数据架构的建立是支撑其他所有架构领域的前提条件。

数据治理愿景：建立以 SAP MDG 为核心的集团级主数据治理平台，实现"One Data, One Truth"（一份数据，一个事实），确保19家法人使用统一的数据标准，为集团合并报表、经营分析、战略决策提供高质量数据基础。

数据治理核心原则：① 单一可信数据源（SOR：System of Record）原则 ② 数据所有权明确原则（每类主数据有指定的Data Owner）③ 数据质量门控原则（主数据进入生产系统前必须通过质量检验）④ 数据生命周期管理原则（创建→审核→启用→停用→归档全程受控）

数据治理组织架构：本报告建议佛照在S/4HANA实施期间同步建立以下数据治理组织：

角色职责建议人员
数据治理委员会（DGC）制定数据策略，审批重大数据标准变更，解决跨部门数据争议李泽华（副总经理，主席）+ 各事业部负责人
数据管理办公室（DMO）日常数据治理运营，制定数据标准，监控数据质量数智化部数据管理骨干（2-3人）
数据所有者（Data Owner）对特定主数据域的质量和完整性负责，审批主数据创建/变更申请物料→研究院；客户→商用事业部；供应商→供应链中心；科目→财务部
数据管家（Data Steward）日常主数据维护操作，在MDG系统中执行审批工作流各业务部门骨干用户（Key Users）

设计维度TO-BE设计方案

③ 财务主数据（Financial Master Data）— SAP MDG-GL

财务主数据的统一是实现19家法人合并报表自动化的基础。当前各法人科目体系差异大，成本中心层级不统一，是合并报表手工操作的根源。

主数据对象TO-BE设计

技术架构（Technology Architecture — TOGAF Phase D）

6.1 L1：技术战略架构

SAP S/4HANA PCE（Private Cloud Edition）是本次佛照技术架构的核心选择。PCE部署模式介于公有云和本地部署之间，由SAP负责底层HANA数据库和S/4HANA应用层的运维，佛照负责业务配置和应用管理，是适合佛照规模（19家法人、综合年交易量超1000万单据）和风险偏好的最优部署方案。

PCE核心特性对佛照的价值
SAP管理底层基础设施佛照无需投入大量IT资源管理HANA数据库；SAP负责数据库补丁、备份、高可用；佛照IT团队专注业务应用
定期强制升级（每季度）自动获得最新功能（包括DRC电子发票更新、法规合规更新）；避免ECC老化17年的重蹈覆辙
BTP扩展平台核心定制开发部署在BTP侧扩展（Side-by-Side Extension），保持S/4HANA核心整洁（Clean Core），不影响季度升级
SAP Cloud ALM项目实施质量门控（Quality Gate）、上线后运维监控、变更管理追踪，全程数字化管理
合规与安全SAP负责ISO 27001、SOC 2认证合规；数据中心在中国境内（数据主权合规）

Clean Core 原则：S/4HANA PCE要求最大化使用标准功能，自定义开发（Z代码）严格控制。现有ECC中的Z程序（Custom Code）必须经过评估：①可用标准功能替代的——废弃；②业务逻辑简单的——按S/4HANA标准ABAP重写；③复杂度高的——迁移至BTP侧扩展；④技术债较高的——纳入风险管理。

SAP系统景观（三套环境）
